Documentación Técnica · Hito 5

Sistema DXN SaaS
Gestión Multisucursal

Documentación técnica del sistema web tipo Micro SaaS para la gestión de cafeterías con múltiples sucursales. Proyecto académico ISI-511, Universidad Privada Franz Tamayo — El Alto, Bolivia.

A · Descripción General

Problema y solución

Las cafeterías con múltiples puntos de venta carecen de herramientas centralizadas para gestionar ventas, inventario y clientes desde una sola plataforma. El control se realiza manualmente o con hojas de cálculo desconectadas entre sucursales, lo que genera pérdida de información, duplicación de trabajo y dificultad para tomar decisiones basadas en datos reales.

Solución SaaS: Sistema web multi-tenant que centraliza en tiempo real las operaciones de ventas, inventario, clientes y reportes para cafeterías con múltiples sucursales. Accesible desde cualquier dispositivo, sin instalaciones, con aislamiento total de datos entre negocios y control de acceso por roles.

B · Módulos Funcionales

Cuatro módulos principales

El sistema está organizado en módulos interconectados que cubren el ciclo operativo completo de una cafetería multisucursal.

🧾 Ventas

Registro de Ventas

Selección de productos del catálogo, cálculo automático del total, descuento de stock en tiempo real, asignación de puntos al cliente y registro del método de pago (efectivo, QR o mixto) en una sola operación.

📦 Inventario

Control de Inventario

Stock diferenciado por sucursal con alertas automáticas al alcanzar el umbral mínimo configurable. Registro de movimientos (entradas, salidas, ajustes) con historial auditado por usuario y fecha.

👥 Clientes

Clientes y Fidelización

Base de clientes con historial de compras y sistema de puntos acumulables por ciclos. Los puntos se pueden canjear en ventas posteriores. Registro de deudas y abonos parciales por cliente.

📊 Reportes

Reportes por Sucursal

Totales de ventas diarias, semanales y mensuales. Productos más vendidos, comparativo entre sucursales y alertas de reposición de stock. Acceso exclusivo para gerente y super administrador.

C · Roles de Usuario

Control de acceso por rol

Cada usuario accede únicamente a las funciones que su rol permite. El aislamiento por negocio_id garantiza que cada tenant opera de forma completamente independiente.

D · Arquitectura del Sistema

Arquitectura de tres capas

El sistema sigue una arquitectura cliente-servidor desacoplada. El frontend y el backend son independientes y se comunican exclusivamente a través de la API REST con autenticación JWT.

1

Frontend

SPA (Single Page Application) con HTML, CSS y JavaScript vanilla. Se sirve como archivos estáticos desde Nginx. Implementa PWA instalable con Service Worker para soporte offline básico.

HTML5 CSS3 JavaScript PWA Nginx
2

Backend

API REST con Node.js y Express. Autenticación con JWT (8 horas), control de roles, rate limiting, helmet.js para seguridad de headers y PM2 en Cluster Mode para alta disponibilidad.

Node.js Express JWT bcrypt PM2 helmet.js
3

Base de Datos

PostgreSQL con 20 tablas relacionales. Aislamiento multi-tenant mediante negocio_id en cada tabla. UUIDs como claves primarias. Migraciones versionadas con scripts SQL.

E · Modelo de Datos

Entidades principales

El modelo de datos tiene 20 tablas en total. Las entidades centrales del sistema son las siguientes:

negocios

  • PK id UUID
  •    nombre TEXT
  •    email_contacto TEXT
  •    creado_en TIMESTAMPTZ

sucursales

  • PK id UUID
  • FK negocio_id
  • FK gerente_id
  •    nombre, activa, bloqueada
  •    es_central BOOLEAN

usuarios

  • PK id UUID
  • FK negocio_id
  • FK sucursal_id
  •    nombre, email, usuario
  •    rol admin|gerente|vendedor

clientes

  • PK id UUID
  • FK negocio_id
  • FK sucursal_id
  •    nombre, telefono
  •    puntos_actuales INTEGER

productos

  • PK id UUID
  • FK negocio_id
  •    nombre, precio, categoria
  •    puntos NUMERIC
  •    es_global, activo BOOLEAN

ventas

  • PK id UUID
  • FK sucursal_id
  • FK usuario_id
  • FK cliente_id
  •    total, estado, metodo_pago

detalle_ventas

  • PK id UUID
  • FK venta_id
  • FK producto_id
  •    cantidad INTEGER
  •    precio_unitario, subtotal
🔒

Aislamiento multi-tenant: Todas las tablas operativas incluyen negocio_id como campo discriminador. Cada consulta de la API filtra automáticamente por el negocio_id extraído del JWT del usuario autenticado, garantizando que un negocio nunca puede acceder a los datos de otro. Este patrón se aplica en los 13 módulos de la API.
